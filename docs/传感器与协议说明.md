# 传感器与协议说明

## 文档状态

- 状态：Authoritative
- 适用对象：处理 IMU/DVL 设备接入、串口识别、协议排障的开发者

## 1. 当前支持的关键设备

当前导航主链重点围绕两类设备：

1. IMU
   - 运行时主线按 WIT Modbus RTU 轮询模型工作
2. DVL
   - 当前以 Hover H1000 为主
   - 常用输出为 PD6/EPD6 风格 ASCII 数据

其他传感器可以存在，但不应模糊这两个主设备对当前主链的核心地位。

## 2. IMU 侧约定

### 2.1 角色

IMU 提供：

- 角速度
- 线加速度
- 姿态相关基础观测
- 高频 propagate 输入

### 2.2 串口与协议

当前运行时按以下假设工作：

- 接口：USB-RS485 或等效串口
- 典型串口配置：`8N1`
- 协议：WIT Modbus RTU
- 设备识别优先看真实串口行为，而不是只看固定设备名

排障时重点核查：

- 供电是否正常
- RS485 A/B 是否接反
- 波特率、从站地址是否与配置一致
- 串口打开后是否真的返回可解析的 Modbus 帧

如果现象是“串口能打开，但始终没有可解析帧”，优先判定为设备/链路问题，不要直接怀疑滤波器。

## 3. DVL 侧约定

### 3.1 角色

DVL 提供：

- 对底速度
- 速度有效位
- 高度或对底距离相关信息
- 低频但关键的 update 输入

### 3.2 Hover H1000 现阶段约定

当前实现默认围绕以下事实构建：

- 主要使用串口输入
- 常见波特率为 `115200`
- 数据格式以 PD6/EPD6 ASCII 语句为主
- 需要区分“串口打开成功”和“真的收到了合法 DVL 语句”

部署与排障时必须满足：

- DVL 的供电、电压与入水安全约束明确
- 不把空气中长时间空转当成正常 bring-up 方式
- 启停流程和现场电源联锁遵守系统文档仓的安全口径

## 4. 设备识别与自动绑定原则

运行时绑定设备时，不以“路径名像不像”作为唯一标准，而是综合：

- `/dev/serial/by-id` 等静态身份
- 串口被动观测到的行为
- 主动探测结果
- 协议是否真的可解析

因此常见状态必须被正确区分：

- `ONLINE`
- `RECONNECTING`
- `MISMATCH`
- 打开成功但无合法帧

这类状态都应该优先进入诊断链，而不是被静默吞掉。

## 5. 协议文档如何维护

今后不再单独维护多个设备协议散文档。维护原则统一为：

- 与 IMU Modbus 有关的变更，直接并入本文 IMU 章节
- 与 DVL PD6/EPD6 有关的变更，直接并入本文 DVL 章节
- 与接线、供电、安全启停有关的关键约束，只保留当前有效版本

如果需要保留详细帧例，只保留能支撑当前运行时实现和排障的最小样例，不再堆历史协议摘录。

## 6. 推荐排障顺序

传感器没起来时，固定按下面顺序判断：

1. 设备供电与线序
2. 串口节点是否存在且稳定
3. 主动/被动探测是否能得到合法协议帧
4. 驱动是否进入 `ONLINE`
5. 样本是否通过预处理门控
6. 之后才看 ESKF 和更高层输出

这样可以避免把大量设备问题误判成融合问题。
