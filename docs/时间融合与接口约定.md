# 时间融合与接口约定

## 文档状态

- 状态：Authoritative
- 适用对象：修改时间基、预处理、ESKF、`NavState` 或日志格式的开发者

## 1. 时间语义目标

导航仓统一时间语义的目标不是“日志看起来顺眼”，而是：

- 保证传感器样本排序稳定
- 让融合 `dt` 可重复
- 让 stale/out-of-order 判定可审计
- 让 replay/compare 能对照真实运行窗口

因此时间相关改动必须同时考虑运行时和证据链，而不是只看单个模块。

## 2. 当前时间基约定

系统默认同时维护两类时间：

1. 单调时间
   - 用于排序、积分、融合和 age 计算
2. 估计实时时间
   - 用于日志、可读展示和跨工具对齐

简单理解：

- 融合和 stale 逻辑看单调时间
- 对外分析和人工阅读可以看估计实时时间

任何模块都不应直接把墙上时钟当成运行时唯一依据。

## 3. 预处理与融合边界

预处理层必须负责把坏样本挡在滤波器外面，至少包括：

- stale
- out-of-order
- 质量位无效
- 与当前设备状态不匹配的样本

ESKF 层负责：

- propagate/update
- 误差状态演化
- 与观测模型一致的融合行为

但 ESKF 不是设备诊断器，也不是最终工程语义发布器。工程语义的最终收口仍在运行时状态发布链。

## 4. `NavState` 契约重点

导航仓输出给外部的核心接口是 `shared::msg::NavState`。

开发时必须默认下列字段属于契约，而不是“便于调试的附带值”：

- 时间戳
- 姿态/位置/速度主载荷
- `valid`
- `stale`
- `degraded`
- `fault_code`
- 传感器相关状态位

如果你改了这些字段的生成逻辑，就等于改了控制侧和 replay 侧对导航的理解，必须同步说明。

## 5. 日志与接口的一致性要求

当前关键日志至少包括：

- `nav_timing.bin`
- `nav_state.bin`
- `nav_events.csv`

一致性要求如下：

- 日志里的时间线要能解释 `NavState` 为什么在某时刻变成 valid/stale/degraded
- replay 输入窗口要能支撑最小状态传播复现
- compare 工具要能对照原始窗口与 replay 结果

不允许出现“运行时语义已变，但日志里没有证据”的漂移。

## 6. 何时可以改时间或融合语义

只有在以下场景才应触碰这类主链语义：

- 真实样本证明当前 stale/timing 逻辑误判
- 预处理门控与协议事实不一致
- replay/compare 发现可重复的时序漂移
- 共享契约需要与真实运行时对齐

禁止因为“看起来更优雅”就大面积重写时间与融合接口。

## 7. 变更闭环

如果本仓改动涉及：

- 时间戳字段
- 预处理门控
- `NavState` 语义
- 日志结构

最低闭环是：

1. 更新本文件
2. 更新系统文档仓对应契约文档
3. 运行受影响的 `nav_core` 构建与最小回归
4. 明确说明哪些消费者已验证，哪些还没验证
