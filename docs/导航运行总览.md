# 导航运行总览

## 文档状态

- 状态：Authoritative
- 适用对象：维护 `nav_core`、设备绑定、日志和 replay 的开发者

## 1. 仓库职责

`Underwater-robot-navigation` 当前负责：

- 驱动 IMU、DVL 等导航关键传感器
- 在板端运行在线守护进程 `uwnav_navd`
- 生成权威 `NavState`
- 输出 `nav_state.bin`、`nav_timing.bin`、`nav_events.csv` 等证据
- 提供最小 replay、compare 和设备诊断工具

它不负责：

- 最终控制 authority
- PWM/STM32 执行
- GCS/UI
- 顶层 supervisor/operator 编排

## 2. 当前在线主链

```text
IMU / DVL
  -> drivers
  -> realtime preprocess
  -> estimator
  -> runtime status gating
  -> shared::msg::NavState
  -> SHM + nav_state.bin + nav_timing.bin + nav_events.csv
  -> control-side consumers + replay / compare
```

这条主链里每一段都承担明确职责：

- `drivers`：把真实设备输入拉进来，同时处理串口、连接状态和协议读写
- `preprocess`：对原始样本做时间、门控、质量筛选，不让坏样本直接进入滤波
- `estimator`：执行 ESKF propagate/update
- `runtime status gating`：把 `valid/stale/degraded/fault_code` 等工程语义写入输出
- `publish/log`：既服务控制侧消费，也保留复盘证据

## 3. 主要目录边界

- `nav_core/`
  - 当前最重要的在线导航运行时
- `apps/`
  - 历史采集脚本与轻量辅助程序，优先级低于 `nav_core`
- `uwnav/`
  - Python 侧工具与离线辅助代码
- `docs/`
  - 本仓中文技术基线文档

在 `nav_core/` 内部，阅读入口建议是：

1. `src/nav_core/app/nav_daemon.cpp`
2. `src/nav_core/app/nav_daemon_runner.cpp`
3. `drivers/`
4. `preprocess/`
5. `estimator/`
6. `io/`
7. `tools/`、`tests/`

## 4. 当前最重要的运行语义

这个仓的工程重点不只是“算出姿态和速度”，而是“带着可信状态语义发布导航结果”。

必须稳定理解的字段包括：

- `valid`
- `stale`
- `degraded`
- `fault_code`
- `sensor_mask`
- `status_flags`

这些字段直接决定控制侧能不能安全消费导航结果，因此不允许把它们退化成“有值就算正常”。

## 5. 运行产物与下游关系

运行时关键产物分三类：

1. 控制侧输入
   - `NavState` SHM
2. 时间与状态证据
   - `nav_state.bin`
   - `nav_timing.bin`
   - `nav_events.csv`
3. replay/诊断辅助
   - incident bundle 输入窗口
   - `uwnav_nav_replay`
   - `replay_compare.py`

这意味着导航仓不仅是“数据生产者”，也是后续故障回放链的起点。

## 6. 设备绑定与重连是主链的一部分

当前实现已经把下面这些行为纳入运行时主链，而不是外围脚本：

- 设备自动识别
- 设备 mismatch
- reconnect
- 无帧、坏帧、超时和 out-of-order 判定
- 健康审查与根因分类

因此，后续修改导航仓时，不允许把这类逻辑降级为“现场自己看日志再猜”。

## 7. 开发时的总原则

- 核心 authority 保持在 C++ `nav_core`
- Python 只做工具、离线处理、日志解析、compare 和辅助手段
- 任何文档、工具、调试脚本都不能反向定义 `NavState` 真实语义
- 如果改动运行时语义，必须同步更新本仓文档与系统级契约文档
