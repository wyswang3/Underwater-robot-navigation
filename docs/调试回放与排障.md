# 调试回放与排障

## 文档状态

- 状态：Authoritative
- 适用对象：做 bench bring-up、真实故障定位、replay/compare 的开发者

## 1. 当前排障主线

导航侧排障优先围绕“证据链”组织，而不是围绕零散命令组织。

主链如下：

```text
uwnav_navd runtime
  -> nav_state.bin / nav_timing.bin / nav_events.csv
  -> merge_robot_timeline.py
  -> incident bundle
  -> uwnav_nav_replay
  -> replay_compare.py
```

核心目标是回答三件事：

1. 问题发生在设备、预处理、融合还是发布环节。
2. 问题发生时有没有结构化时间证据。
3. 后续修复是否能通过 replay/compare 最小回归。

## 2. 常用工具

当前优先级最高的工具是 `nav_core/tools/` 和 `build/bin/` 下的运行时工具：

- `uwnav_navd`
  - 在线导航守护进程
- `uwnav_nav_replay`
  - 最小 NavState 窗口回放
- `uwnav_imu_modbus_probe`
  - 主动探测 IMU 候选串口
- `uwnav_dvl_selftest`
  - DVL 自检
- `merge_robot_timeline.py`
  - 合并 nav/control/telemetry 证据
- `replay_compare.py`
  - 对照原始窗口与 replay 结果
- `usb_serial_snapshot.py`
  - 记录真实串口身份与变化
- `parse_nav_timing.py`
  - 解析时间日志

## 3. 推荐 bring-up 顺序

真实设备 bring-up 固定按下面顺序排查：

1. 看串口节点与静态身份是否稳定
2. 用 probe/selftest 确认设备协议层真的有响应
3. 启动 `uwnav_navd`
4. 确认 `NavState`、`nav_timing.bin`、`nav_state.bin` 是否生成
5. 若状态异常，再看 `nav_events.csv` 与 stderr
6. 需要复盘时，再进入 bundle/replay/compare

这样可以避免直接在“全链路没起来”的情况下盲目改融合代码。

## 4. 常见问题分类

### 4.1 设备侧

- 串口存在但无合法帧
- 错设备被误识别
- 重枚举后路径漂移
- 供电或线序异常

### 4.2 时间与预处理侧

- 样本 stale
- out-of-order
- 质量位无效
- 传感器 age 不符合主链预期

### 4.3 发布与消费侧

- `NavState` 不更新
- 状态位与日志证据不一致
- replay 结果与真实窗口不一致

## 5. 样本收集最低要求

如果 bench 上出现“导航没起来”或“控制侧读不到导航”，至少保留：

- `uwnav_navd` stderr
- `nav_events.csv`
- `nav_state.bin`
- `nav_timing.bin`
- 与之对应的 control/telemetry 时间线样本

如果问题跟 USB 重连有关，再补：

- `/dev/serial/by-id` 快照
- `usb_serial_snapshot.py` 输出

没有这些样本时，不应直接声称已经定位到根因。

## 6. replay/compare 的使用边界

当前 replay 是“最小状态传播回放”，不是完整原始帧级重演。

它适合验证：

- 状态传播语义
- stale/degraded/fault 行为
- 关键窗口回归

它不适合夸大成：

- 完整设备电气复现
- 全原始串口字节级精确仿真

## 7. 排障时的硬约束

- 先收样本，再改核心 C++ 主链
- 一次只改一个小点
- 改完必须做最小可回归验证
- 若问题只是 operator/helper 入口，不要顺手改导航 authority 主链
