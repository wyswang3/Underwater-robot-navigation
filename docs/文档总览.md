# 导航文档总览

## 文档状态

- 状态：Authoritative
- 适用仓库：`Underwater-robot-navigation`
- 目标：把导航侧文档收口为少量高密度中文基线，避免散乱协议笔记、阶段总结和一次性实验记录继续扩散

## 1. 这个仓的文档应该回答什么

导航侧文档只聚焦四件事：

1. 传感器怎样进入 `nav_core` 在线主链。
2. `NavState` 与日志证据怎样生成并提供给控制侧与 replay。
3. 时间、有效性、stale、重连和健康审查语义怎样约束实现。
4. 真实 bench 出问题时，开发者应该怎样排障和收样本。

不在这里展开的内容：

- 顶层多仓启动顺序
- GCS/UI 操作说明
- 系统级 operator 流程

这些统一放在系统文档仓 `UnderwaterRobotSystem/UnderwaterRobotSystem/docs/`。

## 2. 当前保留的六份基线文档

1. `docs/文档总览.md`
   - 入口、阅读顺序、保留规则
2. `docs/导航运行总览.md`
   - 导航主链、模块边界、运行产物
3. `docs/传感器与协议说明.md`
   - IMU/DVL 设备、串口、协议、接线与识别边界
4. `docs/时间融合与接口约定.md`
   - 时间基、融合边界、`NavState`/日志契约
5. `docs/调试回放与排障.md`
   - bring-up、故障定位、replay/compare、样本收集
6. `docs/演进记录与维护约束.md`
   - 历史收口、升级方向、维护禁区

## 3. 推荐阅读顺序

第一次进入这个仓：

1. `README.md`
2. 本文
3. `docs/导航运行总览.md`
4. `nav_core/README.md`
5. `docs/时间融合与接口约定.md`
6. `docs/调试回放与排障.md`

针对具体任务再补：

- 传感器/串口问题：`docs/传感器与协议说明.md`
- 方案延续、阶段背景：`docs/演进记录与维护约束.md`

## 4. 文档收口原则

后续只允许往这六份文档里并入内容，不再新增零散同类文档。判断规则如下：

- 解释运行时主线、状态语义、时间语义：并入 `导航运行总览` 或 `时间融合与接口约定`
- 解释设备型号、协议帧、串口排障：并入 `传感器与协议说明`
- 解释现场排障、日志、replay、incident：并入 `调试回放与排障`
- 解释阶段演进、设计取舍、后续边界：并入 `演进记录与维护约束`

以下内容不再单独留档：

- 一次性命令备忘
- 孤立实验记录
- 与当前 `nav_core` 主线脱节的旧采集流程
- 自动生成中间文件或导出物

## 5. Source Of Truth

导航相关事实优先级固定为：

1. `nav_core/` 真实代码
2. `UnderwaterRobotSystem/shared` 中的共享头文件
3. 本仓六份中文基线文档
4. 系统文档仓中的镜像和 runbook
5. 历史阶段材料

若文档与代码冲突，以运行时代码和共享契约为准，并在对应基线文档中补记录。
