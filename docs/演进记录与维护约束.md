# 演进记录与维护约束

## 文档状态

- 状态：Authoritative
- 适用对象：决定导航侧下一步演进方向或整理历史设计时的开发者

## 1. 当前演进结论

导航仓已经从“先采数、先把滤波跑起来”的原型阶段，进入“运行时语义、重连、回放和诊断可信”的工程阶段。

当前主线不是继续堆功能，而是继续收口：

1. `NavState` 工程语义
2. 设备绑定与 reconnect
3. 时间证据链与 replay/compare
4. `shared/` 单一真源治理

## 2. 已经形成基线的阶段成果

### 2.1 P0/P1 语义收口

已经成为当前基线的内容包括：

- 显式 `valid/stale/degraded/fault_code`
- `NavState` 到控制侧消费链的状态语义收口
- `nav_timing.bin`、`nav_state.bin`、`nav_events.csv`
- 最小 incident bundle 与 replay/compare

### 2.2 设备诊断与健康审查

已进入主线的内容包括：

- 设备自动识别
- reconnect 状态传播
- IMU/DVL 样本拒绝原因审查
- 健康根因分类，例如：
  - `transport_timing`
  - `sensor_input`
  - `estimator_consistency`
  - `estimator_numeric`

## 3. 当前仍需继续收口的方向

- 真实设备重枚举样本
- 更稳定的 replay/compare 节奏一致性
- `shared/` 单一真源与消费者对齐
- 真实 bench 样本驱动的小步修复

## 4. 维护禁区

后续维护时，下面这些方向默认禁止：

- 用 Python 替代在线导航 authority
- 因为代码旧就整体重写 `nav_core`
- 在没有真实样本的情况下大改时间与融合主链
- 把系统级操作文档重新堆回本仓

## 5. 文档维护规则

本仓后续不再保留：

- 阶段性零散总结
- 孤立实验日志
- 与当前主线脱节的旧指南

如果新的设计取舍对后续协作重要，直接更新本文件，不再新增“第 N 轮变更摘要”式文档。

## 6. 修改导航主链前必须先回答的问题

1. 为什么必须动这个点，而不是外围工具或系统文档仓。
2. 这轮是否只动一个小点。
3. 对 `NavState`、日志或 replay 语义有没有影响。
4. 最小验证是什么。
5. 哪些消费者已经验证，哪些还未验证。

答不清这五个问题时，不应进入核心主链改动。
