# IMU & DVL 预处理与融合工程规范（用于在线 C++ 定位算法）

> 适用对象：在线导航 / 定位 C++ 模块（ESKF / KF / INS–DVL）
> 数据来源：IMU（加速度 + 角速度）与 DVL（BI / BE 速度）

---

## 1. IMU 传感器坐标系与物理语义（必须明确）

### 1.1 IMU 传感器原始坐标系定义（厂家语义）

IMU 传感器采用 **右手笛卡尔坐标系**，其轴定义为：

| 轴     | 正方向 | 物理含义    |
| ----- | --- | ------- |
| **X** | 向右  | Right   |
| **Y** | 向前  | Forward |
| **Z** | 向上  | Up      |

即：**RFU（Right–Forward–Up）坐标系**

---

### 1.2 原始加速度数据语义

* 原始加速度单位：**g**
* 静态时（无运动）：

  * 重力模长 ≈ **1 g**
  * Z 轴（向上）读数为 **+1 g**
* 即：IMU 输出的是 **specific force（比力）**

[
f = a - g
]

---

### 1.3 IMU 静态窗诊断结论（已验证）

根据静态窗（前 20s）诊断结果：

* 加速度零均值、低方差
* 角速度零均值、低噪声
* 比力一致性 RMS ≈ **0.012 m/s²**

**结论：**

* IMU 硬件质量与静态数据 **可信**
* 系统漂移的主要来源 **不是 DC 加速度误差**

---

## 2. 核心工程问题：IMU 坐标语义 ≠ 机器人体系语义

### 2.1 坐标不一致是“必须显式处理”的问题

* IMU 使用 **RFU**
* 机器人本体通常使用：

  * **FRD（Forward–Right–Down）**
  * 或等价 Body 坐标定义
* 导航坐标系统一采用：**ENU**

**工程结论：**

> ❌ 不能直接将 IMU 原始数据送入滤波器
> ✅ 必须在 **滤波前** 完成 IMU → Body 的语义统一

---

### 2.2 必须在“滤波前”完成的 IMU 预处理

在进入 ESKF / KF 前，IMU 数据必须完成：

1. **轴重排 / 符号修正（RFU → Body）**
2. **单位换算**

   * g → m/s²
   * deg/s → rad/s
3. **物理语义统一**

   * 明确 yaw 的定义
   * 明确正方向
   * 明确零点参考

> ⚠️ **预处理阶段只做确定性变换，不做状态估计**

---

## 3. Yaw 语义问题：工程事实与最终处理策略（关键修正）

### 3.1 已观测到的工程事实

* 不同 yaw 源之间存在 **近似常数偏差（≈ 90°）**
* 但在整体数据中：

  * yaw 误差 **并非集中在单一常数**
  * 而是近似分布在 **0–180° 的多个区间**
* 表明：

> **yaw 不是“一个可用常数偏置修正”的量，而是语义不闭合的量**

---

### 3.2 重要工程结论（修正原策略）

> **在线 ESKF 中：不再使用任何 IMU yaw 作为观测或约束**

原因：

* yaw 在无磁、无外部航向约束条件下 **弱可观测**
* 现有 IMU yaw 的物理语义 **无法稳定映射到机器人航向**
* 强行使用 yaw 会将错误语义注入滤波器，导致：

  * 水平速度投影错误
  * 轨迹旋转 / 失真
  * 表面 NIS 正常但轨迹物理意义错误

---

### 3.3 最终采用的航向策略（必须写死）

#### ✅ 在线 ESKF 只维护 **相对航向（relative yaw）**

* 初始化：

  ```
  yaw_rel(t0) = 0
  ```
* 传播：

  ```
  yaw_rel(t+dt) = yaw_rel(t) + (ωz - bgz) * dt
  ```
* yaw 仅用于：

  * Body ↔ ENU 的相对旋转
  * 保证速度、加速度在同一内部坐标语义下自洽

#### ❌ 明确禁止：

* 使用 IMU 报文中的 yaw 作为：

  * 初值
  * 观测
  * 修正项

> yaw 在系统中 **不是绝对物理量，而是内部几何中间变量**

❌ 本系统 不引入任何 yaw 相关 bias 状态
因为在无航向观测条件下，yaw bias 与 yaw 本身不可区分，会导致状态不可辨识（unobservable）。

---

## 4. DVL 使用策略：BI 与 BE 的明确分工

### 4.1 为什么 DVL_BE 的水平速度不可信

* DVL_BE 依赖 DVL 内部：

  * 姿态解算
  * 航向估计
* 实测表明：

  * 在二维平面（E/N），BE 速度存在明显系统性误差

---

### 4.2 工程融合策略（已验证）

#### ✅ 水平平面（E/N）：

* **仅使用 DVL_BI（体速度）**
* 在 ESKF 中：

  * 将状态速度投影到 body
  * 与 DVL_BI 直接做更新
* 避免任何 DVL 内部航向对结果的污染

#### ✅ 垂向（U）：

* 可使用 **DVL_BE 的垂向分量**
* 因为：

  * 垂向与 yaw 无关
  * 物理语义更直接
* 可与：

  * IMU 垂向加速度
  * 深度计（若有）
  * 组成独立或弱耦合通道

---

### 4.3 总结性表述（工程 / 论文通用）

> 水平运动采用 DVL 体速度（BI）与 IMU 相对姿态融合，以避免 DVL 内部姿态误差对地速估计的影响；垂向运动则更多依赖 DVL 的地速垂向分量（BE），与 IMU 垂向加速度共同约束，从而在无全局定位条件下获得更稳健的 2.5D 定位结果。

---

## 5. 在线 C++ ESKF 的模块级设计约束

### 5.1 IMU 预处理模块（必须独立）

必须存在独立模块，例如：

```
preprocess/
  imu_preprocess.hpp
  imu_preprocess.cpp
```

职责 **仅限**：

* RFU → Body 坐标变换
* 单位换算
* 静态 bias 估计（启动阶段）
* 可选限幅 / 低通

**禁止：**

* 任何状态估计
* 任何“偷偷修 yaw”的逻辑

---

### 5.2 滤波模块可以假设的输入语义

ESKF / KF 内部 **可以直接假设**：

* `acc_body`：m/s²
* `gyro_body`：rad/s
* `yaw_rel`：内部相对航向（非观测量）
* `v_dvl_bi`：body-frame 速度

---

## 6. 风险总结与强制检查项（Checklist）

上线前必须确认：

* [ ] IMU RFU → Body 坐标变换已启用
* [ ] 单位换算正确（g → m/s²）
* [ ] **滤波器未使用任何 IMU yaw 观测**
* [ ] 水平速度更新 **仅使用 DVL_BI**
* [ ] 垂向速度可使用 DVL_BE
* [ ] yaw 仅由 `(ωz - bgz)` 积分获得

---

## 7. 最终结论（给 C++ 项目和未来维护者）

> 在缺乏全局航向与位置约束的条件下，系统的主要误差来源不是 IMU 或 DVL 的随机噪声，而是 **航向（yaw）的弱可观测性与坐标语义不一致**。
>
> 因此，在线系统应放弃绝对 yaw 的使用，转而采用以起点为参考的相对航向，通过 IMU 角速度传播并配合 DVL 体速度约束，才能获得物理一致、可长期运行的工程级定位结果。
